# 小目標問題分析 與 輸入尺寸 fps 權衡 (2026-06-13)

> 用資料量化 PLAN §1.3 的「無人機在熱影像中很小」風險,並用 KL730 evaluate() 實測
> 不同輸入尺寸的 fps,給出「偵測小目標 vs 即時速度」的工程決策依據。

## 一、目標尺寸分布 (train set, 25,459 張, 40,970 框)

物件像素尺寸 = √(w×h),native 像素。COCO 慣例分桶:tiny<16 / small16-32 / medium32-96 / large>96。

| 類別 | 框數 | 中位尺寸 | p10 | p90 | tiny<16 | small16-32 | med32-96 | large>96 |
|---|---|---|---|---|---|---|---|---|
| **uav** | 23,452 | 32px | 10 | 55 | **22%** | **29%** | 48% | 2% |
| person | 8,533 | 15px | 11 | 22 | 58% | 42% | 0% | 0% |
| vehicle | 5,355 | 47px | 23 | 85 | 3% | 21% | 72% | 5% |
| bicycle | 3,630 | 27px | 16 | 46 | 9% | 58% | 33% | 0% |

## 二、小目標問題有多嚴重

- **UAV(核心任務):50.9% 的目標 < 32px,22.0% < 16px。**
- person 更極端:58% < 16px(空對地 60–130m 飛高,人在熱影像中就是幾個像素的亮點)。
- → 這不是邊角案例,而是**多數目標都很小**,小目標能力直接決定系統可用性。

### 為什麼這跟 head 的 stride 有關

YOLOX 預設三個偵測頭 P3/P4/P5,stride 8/16/32。stride-8 的 P3 負責最小目標,
其可靠分辨下限約 **16–24px**。所以:
- 640 輸入下,**< 16px 的目標(UAV 佔 22%、person 佔 58%)落在 P3 下限以下,容易漏**。
- 對策:① 提高輸入解析度(把小目標放大);② 加 **P2 head**(stride-4,下限可到 ~8px);③ tiling 切片。

## 三、輸入尺寸 → KL730 fps 實測 (evaluate / ip_evaluator)

模型:YOLOX-s(zoo 權重),KL730 INT8 模擬。

| 輸入尺寸 | KL730 fps | CPU fallback node | 等效:16px native 目標放大到 | 評語 |
|---|---|---|---|---|
| **640×640** | **43.2** | 0 | ~16px (原尺寸,640×512 native) | 最快,但漏 <16px 小目標 |
| **896×896** | **21.3** | 0 | ~22px | **平衡點**:仍近即時,小目標進入 P3 可靠區 |
| **1280×1280** | **10.1** | 0 | ~32px | 小目標最佳,但跌破 25fps 即時線 |
| 640 + P2 head | ~25–30 (估算) | 待驗 | P2 下限 ~8px | 不放大全圖、只補小目標頭;需 config 改動驗證 |

> fps 與像素數約成反比(640→1280 = 4× 像素 → ~1/4 fps),三個實測點吻合此關係。
> 全部 **零 CPU fallback** = 整張圖都在 NPU 上跑,沒有 operator 退回 CPU 拖慢。

## 四、工程結論(給部署決策)

1. **若要求 ≥25 fps 即時** → 640(43fps)或 896(21fps,略低於 25 但接近)。
2. **若小目標(遠距 UAV)是硬需求** → 896 是 sweet spot;1280 留給「慢速但要看清」的場景。
3. **最佳 CP 值候選:640 + P2 head** —— 不付全圖放大的速度代價,只補強小目標頭,估 25–30fps。值得在訓練後做一次 PoC 驗證(改 config + 重新 evaluate)。
4. 最終取捨要等**真實偵測距離需求**(無人機要在幾公尺外認出目標)確定後拍板 —— 這個數值目前未定義(見 CONTEXT「Flagged ambiguities」)。

## 重現方式

```bash
# 目標尺寸分布: 讀 data/coco/annotations/train.json 統計 √(w×h) 分桶 (純 CPU)
# fps: 匯出各尺寸 ONNX → toolchain evaluate()
make export-onnx                      # 或對 zoo 權重以 --shape 896 896 / 1280 1280 匯出
# docker run ... ktc.ModelConfig(...,"730").evaluate() → 讀 ip_eval/fps 與 cpu_node
```
