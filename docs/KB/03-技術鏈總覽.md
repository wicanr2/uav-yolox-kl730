# L3 · 技術鏈總覽 — 從資料到無人機的七站

> 這一層是整個專案的**脊椎**。看懂這張圖,你就知道每個檔案、每個 `make` 指令在哪一站、做什麼。

## 全景圖

```
        【訓練世界 — PC / 雲端 GPU】                    【部署世界 — KL730 晶片】
┌─────────────────────────────────────────┐   ┌──────────────────────────────────┐
│ 1.蒐集 → 2.轉換 → 3.訓練 → 4.匯出ONNX     │   │ 5.量化+編譯 → 6.後處理 → 7.部署    │
│  資料集    COCO     YOLOX    .onnx        │──▶│  NEF          decode+NMS   機上跑  │
└─────────────────────────────────────────┘   └──────────────────────────────────┘
              ↑ 橋樑就是 ONNX(通用格式)→ Kneron toolchain 接手 ↑
```

## 七站逐站拆解

### 站 1 · 資料蒐集
- **做什麼**:抓三個熱影像資料集 — TU2U(空對空無人機)、HIT-UAV(空對地人/車)、Anti-UAV410(地對空無人機,擴充)。
- **檔案/指令**:`datasets/download.sh` / `make data-download`
- **產出**:`data/raw/` 下三個原始資料集。

### 站 2 · 資料轉換
- **做什麼**:三個來源格式都不同(YOLO txt、偽 COCO、tracking json),全部轉成統一的 **COCO 4 類**(uav/person/vehicle/bicycle)再合併。
- **檔案/指令**:`datasets/convert_*.py` + `merge_coco.py` / `make data-convert`
- **產出**:`data/coco/annotations/{train,val,test}.json`(train 約 2.5 萬張)。
- **關鍵**:HIT-UAV 的「偽 COCO」有三個格式陷阱(見檔案註解);Anti-UAV410 抽幀避免灌水。

### 站 3 · 訓練
- **做什麼**:用資料把 YOLOX-s 教會認 4 類。
- **檔案/指令**:`training/configs/yolox_s_uav2uav.py` + `make train`(雲端)/ `make train-smoke`(本機驗證)
- **產出**:`work_dirs/.../latest.pth`(模型權重)。
- **在哪跑**:本機只有 CPU,正式訓練在 **Kaggle 免費 GPU**(見 `kaggle/`)。本機只能跑 smoke(證明管線會動)。

### 站 4 · 匯出 ONNX
- **做什麼**:把 PyTorch 權重轉成通用格式 ONNX — 這是跨進「晶片世界」的橋。
- **檔案/指令**:Kneron 的 `pytorch2onnx_kneron.py` / `make export-onnx`
- **產出**:`artifacts/*.onnx`(opset 11,`--skip-postprocess` = 只到 raw 輸出,不含 decode)。

### 站 5 · 量化 + 編譯(整條鏈最關鍵的一站)
- **做什麼**:ONNX → 優化 → **INT8 量化** → 編譯成 KL730 專屬可執行檔 **NEF**。
- **檔案/指令**:`conversion/convert_to_nef.py` / `make nef`(在 Kneron toolchain docker 內)
- **產出**:`kneron_flow/models_730.nef`(約 9MB)。
- **關鍵教訓**:量化範圍估計法必須用 `mmse`,不能用預設 `percentage=1.0`(對 raw-logit 會崩潰成常數)。這是我們查 7 輪才找到的根因 → L5。
- **內建驗收**:逐 channel 比對 float vs NEF 的相關性(`--check`),抓量化退化。

### 站 6 · 後處理(decode + NMS)
- **做什麼**:NEF 吐的是 9 個 raw 張量,要 decode 成「框」,再用 NMS 去掉重複框。**這段在 KL730 上跑的是 A55 CPU,不是 NPU**。
- **檔案**:`firmware/detect/postprocess.py`(純 numpy,本機已單元測試)
- **為什麼分開**:decode/NMS 含 NPU 不擅長的運算,慣例交給 CPU;NPU 只算到 raw 張量。

### 站 7 · 部署
- **做什麼**:在實體 KL730 上用 Kneron PLUS 載入 NEF、餵影像、跑推論、接後處理,結果經 MAVLink/RTSP 下傳。
- **檔案**:`firmware/detect/detector.py`(窄介面 `infer(frame)→[框]`,sim/board 雙 backend)
- **狀態**:sim backend 本機可跑(模擬器);board backend 待實體硬體。

## 貫穿全鏈的「驗證」(很重要的觀念)

好的工程不是「跑完就好」,而是**每一站都有辦法驗證對不對**:

| 驗證 | 在哪站 | 檔案/指令 | 抓什麼 |
|---|---|---|---|
| 資料能被載入 | 2→3 之間 | `make smoke` | json/路徑/類別對不對 |
| 訓練管線會動 | 3 | `make train-smoke` | loss 會降、checkpoint 會生 |
| 量化沒崩潰 | 5 | `convert_to_nef.py --check` | float vs NEF 逐 channel 相關性 |
| 量化精度達標 | 5→6 | `conversion/eval_map.py` | mAP 掉幅 <3%(待正式模型) |
| 端到端能出框 | 6 | `conversion/run_nef_demo.py` | 真實熱影像畫出框 |
